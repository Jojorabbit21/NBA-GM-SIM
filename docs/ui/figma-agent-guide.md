# 디자인 에이전트 & 스킬 사용 가이드

피그마 디자인을 코드로 연결하는 두 가지 도구의 사용법.

---

## 사전 조건

- **Figma 데스크탑 앱** 실행 중
- **Figma MCP** Claude Code에 연결됨
- 작업할 피그마 파일이 앱에서 열려 있음

---

## 도구 1: `/design-sync` — 토큰 동기화

피그마 Variables를 읽어 `tailwind.config.ts`와 CSS 변수에 반영한다.

### 사용법

1. 피그마에서 Variables가 적용된 아무 노드나 선택
2. Claude Code에서 실행:

```
/design-sync
```

### 언제 쓰나

- 피그마에서 색상값을 변경했을 때
- 새 토큰(spacing, radius 등)을 추가했을 때
- 코드의 토큰이 피그마와 맞는지 검증하고 싶을 때

### 출력

- 추가/변경/삭제된 토큰 목록
- 확인이 필요한 예외 항목
- 업데이트된 파일 경로

---

## 도구 2: `@design-system` — 디자인 구현 에이전트

피그마를 직접 읽어 컴포넌트를 구현하는 전담 에이전트.

### 사용법

메시지 앞에 `@design-system`을 붙인다:

```
@design-system 지금 피그마에서 선택한 사이드바 버튼 컴포넌트 구현해줘
```

```
@design-system 홈 화면 레이아웃을 피그마 디자인에 맞게 수정해줘
```

```
@design-system 현재 선택한 카드 컴포넌트의 hover 상태가 코드에 올바르게 구현됐는지 확인해줘
```

### 언제 쓰나

| 상황 | 예시 |
|---|---|
| 새 컴포넌트 구현 | "피그마에서 선택한 모달 컴포넌트 만들어줘" |
| 기존 컴포넌트 수정 | "디자인에 맞게 버튼 스타일 업데이트해줘" |
| 디자인 원칙 검증 | "이 파일이 현재 디자인 시스템을 잘 따르고 있는지 확인해줘" |
| 예외 규칙 파악 | "30개 팀 중 다른 토큰을 쓰는 팀을 찾아서 정리해줘" |

### 에이전트 동작 방식

1. 피그마에서 대상 노드의 디자인 컨텍스트와 Variables를 읽음
2. 현재 코드베이스에서 관련 파일 파악
3. 피그마 원칙 기준으로 구현 또는 수정
4. 구현 후 자체 검증

> **중요**: 에이전트는 항상 피그마를 먼저 읽는다.
> 기억에 의존한 값이나 추측으로 구현하지 않는다.

---

## 새 디자인 시스템으로 전환할 때

디자인을 전면 교체한 경우 순서:

```
1. 피그마에서 새 Variables / 컴포넌트 정의 완료
2. /design-sync 실행 → 토큰 코드 반영
3. @design-system에게 컴포넌트별 구현 요청
```

에이전트 파일(`.claude/agents/design-system.md`)은 **수정할 필요 없다.**
방법론만 담겨 있어 피그마가 바뀌면 자동으로 새 디자인 기준으로 동작한다.

## 트러블슈팅

| 증상 | 원인 | 해결 |
|---|---|---|
| `{}` 반환 | 피그마에서 노드 미선택 | 아무 노드나 클릭 후 재시도 |
| `No session found` | MCP 연결 끊김 | 피그마 앱 재시작 후 재시도 |
| `Nothing is selected` | 피그마가 백그라운드 | 피그마 앱 클릭해서 포커스 |
| 값이 구식 | 피그마 미열람 | 에이전트에게 "피그마 다시 읽어줘" 요청 |
